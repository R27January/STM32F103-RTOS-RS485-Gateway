# FreeRTOS Design

This document describes the RTOS task architecture and synchronization design of this project.

## Overview

This project uses FreeRTOS to separate data acquisition, display, communication, and logging into independent tasks.

The main purpose of using FreeRTOS is to make the system easier to maintain and closer to a real embedded engineering structure.

## Task List

| Task          | Function                                         |
| ------------- | ------------------------------------------------ |
| `SensorTask`  | Periodically acquires ADC data                   |
| `DisplayTask` | Displays sensor data and system status on OLED   |
| `CommTask`    | Handles RS485 communication and protocol parsing |
| `LogTask`     | Writes log records into W25Q64 external Flash    |
| `DefaultTask` | Reserved system task                             |

## Task Responsibilities

### SensorTask

`SensorTask` is responsible for periodic ADC sampling.

Main responsibilities:

* Read ADC value
* Update sample counter
* Package data into `SensorData_t`
* Send sensor data to display and communication queues

Data flow:

```text
SensorTask
↓
DisplayDataQueue
↓
DisplayTask
```

```text
SensorTask
↓
CommDataQueue
↓
CommTask
```

---

### DisplayTask

`DisplayTask` is responsible for OLED display.

Main responsibilities:

* Receive latest sensor data from `DisplayDataQueue`
* Read system status from `SystemEventGroup`
* Display ADC value, sample counter and communication status

Displayed communication status:

| Status      | Meaning                             |
| ----------- | ----------------------------------- |
| `IDLE`      | No active communication status      |
| `COMM OK`   | Recent valid communication received |
| `PROTO ERR` | Protocol error occurred             |

---

### CommTask

`CommTask` is responsible for RS485 communication.

Main responsibilities:

* Wait for UART DMA + IDLE reception notification
* Copy received DMA data into local buffer
* Check frame length
* Verify protocol and CRC16-Modbus
* Process command type
* Send response through RS485
* Update communication status event bits
* Restart communication timeout software timer

Current supported commands:

* Query current sensor data
* Query historical log data

The current communication reception flow is:

```text
UART receives data by DMA
↓
IDLE event occurs
↓
HAL fixed function records received length
↓
Task notification wakes up CommTask
↓
CommTask parses protocol frame
↓
CommTask sends RS485 response
↓
Restart DMA reception
```

---

### LogTask

`LogTask` is responsible for writing log data into W25Q64 external Flash.

Main responsibilities:

* Wait for log records from `LogQueue`
* Calculate log storage address
* Write log record to W25Q64
* Keep communication and Flash writing separated

Log data is generated by `CommTask` and written by `LogTask`.

Data flow:

```text
CommTask
↓
LogQueue
↓
LogTask
↓
W25Q64
```

## RTOS Mechanisms

### Queue

Queues are used to transfer structured data between tasks.

Used queues:

| Queue              | Function                                   |
| ------------------ | ------------------------------------------ |
| `DisplayDataQueue` | Transfers sensor data to `DisplayTask`     |
| `CommDataQueue`    | Transfers latest sensor data to `CommTask` |
| `LogQueue`         | Transfers log records to `LogTask`         |

Reason for using queues:

* Decouple producer and consumer tasks
* Avoid direct global data dependency
* Improve task structure clarity

---

### Mutex

Mutexes are used to protect shared resources.

Used mutexes:

| Mutex       | Protected Resource         |
| ----------- | -------------------------- |
| `OLEDMutex` | OLED display               |
| `UARTMutex` | UART / RS485 communication |

Reason for using mutexes:

* Prevent multiple tasks from accessing the same hardware resource at the same time
* Avoid display or communication conflicts

---

### Task Notification

Task notification is used to wake up `CommTask` after UART DMA reception.

Reason for using task notification:

* Lighter than queue or semaphore
* Suitable for waking up a specific task
* Works well with UART DMA + IDLE reception

Current usage:

```text
HAL_UARTEx_RxEventCallback
↓
vTaskNotifyGiveFromISR
↓
ulTaskNotifyTake in CommTask
```

---

### Event Group

Event group is used to manage system communication status.

Used event bits:

| Event Bit            | Meaning                                  |
| -------------------- | ---------------------------------------- |
| `EVENT_COMM_OK`      | A valid communication frame was received |
| `EVENT_PROTOCOL_ERR` | A protocol error occurred                |

Reason for using event group:

* System status can be represented by bit flags
* Multiple tasks can read the same status
* Suitable for communication state display and diagnosis

---

### Software Timer

Software timer is used for communication timeout handling.

Current timer:

| Timer              | Function                                           |
| ------------------ | -------------------------------------------------- |
| `CommTimeoutTimer` | Clears `EVENT_COMM_OK` after communication timeout |

Current behavior:

```text
Valid communication received
↓
Set EVENT_COMM_OK
↓
Restart CommTimeoutTimer
↓
No new valid communication for 3 seconds
↓
Timer callback clears EVENT_COMM_OK
↓
Display status returns to IDLE
```

Reason for using software timer:

* Avoid creating an extra task just for timeout counting
* Keep communication timeout logic simple
* Make status recovery automatic

## System Data Flow

Overall system flow:

```text
ADC
↓
SensorTask
↓
DisplayDataQueue ──→ DisplayTask ──→ OLED
↓
CommDataQueue
↓
CommTask ← UART DMA + IDLE + Task Notification
↓
RS485 Response
↓
LogQueue
↓
LogTask
↓
W25Q64
```

## Current RTOS Baseline

The current version has completed:

* Task-based system separation
* Queue-based data transfer
* Mutex-based resource protection
* UART DMA + IDLE reception
* Task notification based communication wake-up
* Event group based status management
* Software timer based communication timeout recovery

## Future RTOS Improvements

Planned improvements:

* Add `ModbusTask` for downstream Modbus RTU device polling
* Add communication retry mechanism
* Add device online/offline status management
* Add communication error counters
* Add watchdog task or watchdog refresh strategy
* Add more complete log management
